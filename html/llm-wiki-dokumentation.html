01 · Das Problem

Wissen geht verloren

Unser Team produziert täglich wertvolles Wissen: DAX-Patterns in Power BI, Architekturentscheidungen in Snowflake, Feature-Engineering-Strategien für ML-Modelle, Troubleshooting-Lösungen. Aber dieses Wissen steckt in Chat-Verläufen, Köpfen und verstreuten Notizen.

Heute: Wissen geht verloren

  • DAX-Lösung im Chat gefunden, drei Monate später nochmal suchen
  • Neuer Kollege fragt "Warum N-HiTS statt Prophet?" Niemand erinnert sich an alle Details
  • Gleiche Probleme werden mehrfach gelöst, weil Lösungen nicht dokumentiert sind
  • RAG/ChatGPT findet Dokumente, synthetisiert aber nicht

Mit LLM Wiki: Wissen akkumuliert

  • Jede Lösung wird einmal dokumentiert und ist sofort auffindbar
  • Architekturentscheidungen mit Kontext und Begründung nachschlagbar
  • Querverweise zeigen wie Themen zusammenhängen
  • Wissen wird synthetisiert, nicht nur abgelegt
02 · Die Kernidee

Das LLM pflegt das Wiki.
Du pflegst das Wissen.

Das LLM Wiki dreht die klassische Dokumentation um. Statt dass Menschen Wikis schreiben und pflegen (und irgendwann aufgeben, weil der Aufwand zu gross wird), übernimmt ein LLM-Agent die gesamte Pflege: Zusammenfassen, Vernetzen, Aktualisieren, Widersprüche erkennen.

Der Mensch macht das, was er am besten kann: Quellen auswählen, Fragen stellen, Richtung geben. Das LLM macht den Rest: die Arbeit, die niemand machen will, aber die ein Wiki erst wertvoll macht.

Der Unterschied zu RAG und ChatGPT

RAG / Datei-Upload: Du lädst Dokumente hoch, das LLM sucht bei jeder Frage von Neuem die relevanten Stellen. Nichts wird aufgebaut. Bei der nächsten Frage fängt es wieder bei null an.

LLM Wiki: Das LLM liest eine Quelle einmal, extrahiert das Wissen, und integriert es in bestehende Seiten. Die Synthese ist bereits gemacht. Die Querverweise stehen. Beim nächsten Ingest wird das Wissen reicher, nicht neu aufgebaut.

Der Effekt: Mit jedem Ingest werden Antworten genauer und einheitlicher. Das Wiki konsolidiert Wissen aus vielen Quellen auf einer Seite. Widersprüche werden erkannt und markiert. Wer eine Frage stellt, bekommt eine Antwort die auf dem gesamten akkumulierten Wissen basiert, nicht auf zufällig gefundenen Textfragmenten.

03 · Wie es funktioniert

Drei Operationen, ein Kreislauf

Das Wiki wächst durch drei sich wiederholende Operationen. Jede davon wird vom LLM-Agenten ausgeführt, gesteuert durch den Maintainer.

Ingest
Quelle verarbeiten
Neues Paper, Artikel oder Notiz → LLM liest, erstellt eine Zusammenfassung, aktualisiert bestehende Seiten, setzt Querverweise, markiert Widersprüche. Ein Ingest berührt 5–15 Wiki-Seiten.
Query
Fragen stellen
Eine Frage an das Wiki. Das LLM sucht relevante Seiten, synthetisiert eine Antwort. Wertvolle Antworten werden als neue Seiten gespeichert. So wächst das Wiki auch durch Fragen.
Lint
Qualität prüfen
Regelmässiger Gesundheitscheck: Widersprüche, veraltete Seiten, verwaiste Einträge, fehlende Querverweise, Lücken die gefüllt werden könnten.
04 · Architektur

Drei Schichten

Das Wiki besteht aus drei klar getrennten Schichten. Jede hat eine eigene Aufgabe und eigene Regeln.

01

Raw Sources

Die Originalquellen: Papers, Artikel, Experiment-Logs, Meeting-Notes. Liegen im Ordner raw/ und werden nie verändert. Sie sind die Source of Truth. Das LLM liest sie, schreibt sie aber nicht.

02

Das Wiki

Ein Ordner mit Markdown-Dateien im Ordner wiki/. Zusammenfassungen, Entity-Seiten, Konzept-Erklärungen, Architekturentscheidungen, Runbooks. Das LLM besitzt diese Schicht vollständig: es erstellt, aktualisiert und verlinkt alle Seiten. Der Mensch liest.

03

Das Schema

Die Datei CLAUDE.md im Root. Die Konfiguration für den LLM-Agenten. Definiert Seitentypen, Namenskonventionen, Workflows und domänen-spezifische Regeln. Mensch und LLM entwickeln das Schema gemeinsam weiter.

05 · Wiki-Struktur

Fünf Seitentypen

Jede Wiki-Seite gehört zu einem Typ. Jeder Typ beantwortet eine andere Frage.

E
Entity

Ein System, Modell oder Tool. "Was ist N-HiTS?"

C
Concept

Ein Pattern oder Technik. "Wie funktioniert SHAP?"

D
Decision

Eine Architektur­entscheidung. "Warum N-HiTS statt Prophet?"

S
Source

Zusammenfassung einer Quelle. "Was steht im Paper?"

R
Runbook

Operative Anleitung. "Forecast-DAG hängt. Was tun?"

Dazu kommen zwei Navigations-Dateien: Index (Inhaltsverzeichnis aller Seiten) und Log (chronologisches Protokoll aller Aktivitäten). Der Index ist der Einstiegspunkt für das LLM und für den Menschen.

06 · Wer macht was

Maintainer und Konsumenten

Wiki-Maintainer (Human in the Loop)

1–2 Personen
  • Quellen auswählen und in raw/ ablegen
  • Claude Code starten und "Ingest" auslösen
  • Ergebnisse prüfen und committen
  • Fragen stellen, Seiten generieren lassen
  • Regelmässig "Lint" durchführen
  • Arbeitet in: VS Code + Claude Code + Git

Wiki-Konsument

Ganzes Team
  • Website aufrufen im Browser. Kein Setup nötig
  • Volltextsuche nutzen
  • Seiten durchklicken, Querverweise folgen
  • Quellen an den Maintainer schicken für Ingest
  • Optional: selbst eine Markdown-Seite beitragen
  • Arbeitet in: Browser (oder Obsidian)
07 · Tool Stack

Welche Tools und warum

Kern

Claude Code

Der LLM-Agent im Terminal. Liest und schreibt Wiki-Dateien, führt Ingests durch, beantwortet Fragen gegen das Wiki. Nutzt die CLAUDE.md als Arbeitsanweisung.

Editor

Obsidian

Lokaler Markdown-Editor mit Graph View. Zeigt die Vernetzung aller Wiki-Seiten als interaktives Netzwerk. Für den Maintainer als Live-Preview. Kostenlos, alles lokal.

Version

Git + GitHub

Versionierung jeder Änderung. Sicherheitsnetz: jede LLM-Änderung kann geprüft und zurückgerollt werden. GitHub als zentraler Speicherort für das Team.

Website

MkDocs + GitHub Pages

Rendert die Markdown-Dateien automatisch als durchsuchbare Website. Bei jedem Push wird die Seite neu gebaut. Das Team braucht nur einen Browser. Kein Setup, kein Git, kein Obsidian.

IDE

VS Code

Entwicklungsumgebung für den Maintainer. Zwei Terminals nebeneinander: Terminal 1 für Claude Code, Terminal 2 für Git-Befehle.

08 · Datensicherheit

Was lokal bleibt, was rausgeht

Das Wiki enthält technisches Handwerkswissen, keine sensiblen Geschäftsdaten. Keine Kundennamen, keine Umsatzzahlen, keine internen Organisationsdetails. Trotzdem ist Transparenz über den Datenfluss wichtig.

L
Bleibt lokal

Wiki-Dateien, Git-Repo, Obsidian. Alles auf dem Rechner des Maintainers.

E
Verschlüsselt zu Anthropic

Dateien die Claude Code liest, werden per TLS an die API gesendet. Kein Training bei deaktiviertem Opt-in. 30 Tage Retention.

W
Öffentlich auf GitHub

Die Website ist öffentlich einsehbar. Nur technisches Wissen publizieren. Keine Geschäftsgeheimnisse.

Enterprise-Lösung für sensible Daten

Für den PoC nutzen wir einen Consumer-Account mit deaktiviertem Training. Für produktiven Einsatz mit sensiblen Geschäftsdaten bietet Anthropic den Enterprise-Plan: vertragliches Data Processing Agreement (DPA), konfigurierbares Data Retention, optional Zero Data Retention (ZDR) bei dem keine Daten gespeichert werden, und Hosting über AWS Bedrock oder Google Vertex AI innerhalb der eigenen Cloud-Infrastruktur.

11 · Vision

Vom ML-Wiki zum BI-Team-Wiki

Das ML-Wiki ist nur der Anfang. Das gleiche Pattern funktioniert für alle Wissensbereiche des Teams oder andere Bereiche und Abteilungen in der Pistor AG. Ein Wiki, das das gesamte akkumulierte Wissen strukturiert und durchsuchbar macht.

PBI

Power BI & DAX

DAX-Patterns (CROSSFILTER, TREATAS, Disconnected Slicers), Calculation Groups, UDFs, Report-Dokumentation, Semantic Model Best Practices

DE

Data Engineering

Snowflake Data Vault 2.0, dbt Konventionen, Airflow DAGs, Key-Pair Auth, Naming Conventions, Pipeline-Troubleshooting

ML

Machine Learning

Forecasting-Modelle, Feature Engineering, Hyperparameter-Tuning, SHAP, Experiment-Tracking, Foundation Models

OPS

Operations & Runbooks

Monitoring-Setup, Gateway-Troubleshooting, Refresh-Fehler, Deployment-Checklisten, Onboarding neuer Teammitglieder

Warum das funktioniert

Das Problem war nie die Kompetenz des Teams. Es war die Pflege. Wikis sterben nicht weil niemand schreiben kann, sondern weil der Wartungsaufwand exponentiell wächst: Querverweise aktualisieren, Zusammenfassungen pflegen, Widersprüche erkennen, Index nachführen. Das LLM erledigt genau diese Arbeit. Zuverlässig, bei jedem Ingest, über alle Seiten hinweg. Die Kosten der Pflege sinken auf null.

Das Team macht das Denken. Das LLM macht das Housekeeping.
